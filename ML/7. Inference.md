




# ML — Inference

---

## What Inference Is

Running a trained model on new inputs to produce predictions.  
No gradients. No weight updates. Focus: **speed, memory, cost**.

---

## 1. Inference vs Training



|              | Training      | Inference                 |
| ------------ | ------------- | ------------------------- |
| Gradients    | Yes           | No (`torch.no_grad()`)    |
| Batch size   | Large         | Small (often 1)           |
| Dropout / BN | Active        | Disabled                  |
| Precision    | FP32 / BF16   | INT8 / FP16 / BF16        |
| Goal         | Minimize loss | Minimize latency / memory |

---

## 2. Key Metrics

| Metric         | Meaning                             |
| -------------- | ----------------------------------- |
| Latency        | Time to process one request (ms)    |
| Throughput     | Requests or tokens per second       |
| Memory (VRAM)  | Model size + activations + KV cache |
| Cost per token | For LLMs at scale                   |

Latency vs throughput are often at odds — batching improves throughput but increases latency.

---

## 3. Quantization

Reduce model weight precision to save memory and speed up compute.

|Format|Bits|Memory vs FP32|Quality Loss|
|---|---|---|---|
|FP32|32|1×|None (baseline)|
|FP16 / BF16|16|0.5×|Minimal|
|INT8|8|0.25×|Small|
|INT4|4|0.125×|Noticeable|
|GPTQ / AWQ|4-bit|~0.125×|Optimized 4-bit for LLMs|

**Post-training quantization (PTQ):** quantize after training, no retraining needed.  
**Quantization-aware training (QAT):** simulate quantization during training — better quality.

---

## 4. Pruning

Remove weights or neurons that contribute little useful output.

| Type              | What is removed                      |
| ----------------- | ------------------------------------ |
| Unstructured      | Individual weights (sparse matrix)   |
| Structured        | Entire heads, filters, or layers     |
| Magnitude pruning | Remove weights with smallest `\|w\|` |

Structured pruning → actual speedup. Unstructured → smaller file, but needs sparse hardware.

---

## 5. Knowledge Distillation

Train a small **student** model to mimic a large **teacher** model.

```
Loss = α · CrossEntropy(y, ŷ_student) + (1-α) · KL(ŷ_teacher ‖ ŷ_student)
```

|Term|Role|
|---|---|
|Teacher|Large, pretrained model|
|Student|Small, fast model being trained|
|Soft labels|Teacher's output distribution (richer signal than hard labels)|
|Temperature T|Soften distributions: `softmax(z/T)` — higher T = softer|

## 6. LLM-Specific

### KV Cache

During autoregressive generation, attention keys and values for past tokens are cached — not recomputed.

- Saves compute on each new token
- Grows with sequence length — main memory bottleneck at inference

### Decoding Strategies

|Strategy|How|Use|
|---|---|---|
|Greedy|Always pick highest probability token|Fast, deterministic, repetitive|
|Beam search|Keep top-k sequences at each step|Translation, structured output|
|Top-k sampling|Sample from top k tokens|Creative generation|
|Top-p (nucleus)|Sample from tokens summing to prob p|More adaptive than top-k|
|Temperature|Scale logits: `softmax(z/T)`|T<1 = sharper, T>1 = more random|

### Batching for LLMs

|Method|Idea|
|---|---|
|Static batching|Wait for full batch — wastes GPU on short sequences|
|Dynamic / continuous batching|Add new requests mid-flight — standard in production|

## 8. Latency Optimization Checklist

```
□ Use no_grad() / eval() mode
□ Use BF16 or FP16
□ Quantize if memory-constrained (INT8/INT4)
□ Enable KV cache (LLMs)
□ Batch requests where possible
□ Use continuous batching for LLM servers
□ Profile — find the actual bottleneck before optimizing
```
